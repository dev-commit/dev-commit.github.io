Squash Commit

  • Хорошая практика сквошить коммиты в один, чтобы на задачку был один коммит
  • Вообще засквошить можно когда угодно, потом форс-пушем долить, но после того как проверен PR этого уже делать не нужно будет, все правки после ревью доливаются отдельными коммитами
git checkout develop # перейти в ветку develop git fetch origin # подтянуть последние изменени для develop git pull # взять последние изменени для develop git checkout feature # перейти в ветку feature # первый вариант (первый этап делать не нужно) git reset --soft origin/develop # берётся ветка из удаленного репозитория # второй вариант git merge-base dev HEAD # номер коммита к которому нужно откатиться git reset HashCommit # откатывает всё до момента откалывания от develop git add . # индексация изменений git commit -m 'Message' # коммит git push origin HEAD # push git push --force origin HEAD # push force (скорее всего именно force)

Отмена изменений

git checkout -- file.txt # Отмена изменений файла git reset HEAD file.txt # Отмена индексации файла git commit --amend # Изменение последнего коммита # Все три команды вместе дают один коммит - второй коммит заменяет результат первого git commit -m 'Message' git add forgotten_file git commit --amend git checkout . # восстановить все git checkout HashCommit # вернуть до состояния этого коммита # Просмотреть, а затем удалить непроиндексированные файлы и папки: git clean -nd git clean -fd # Просмотреть, а затем удалить ВСЕ непроиндексированные файлы и папки, включая игнорируемые Git-ом git clean -ndx git clean -fdx

Перенос репозитория

clone

  • Команда используется для полной синхронизации содержимого одного удалённого репозитория с другим
  • Она копирует все ветки, теги и рефы из локального репозитория в указанный удалённый репозиторий, полностью заменяя его содержимое
  • Это создаёт "зеркальный" локальный репозиторий, который хранит все ветки и теги
git clone --mirror UrlИсходногоРепозитория

push

  • Команда перенесёт все данные из локального зеркального репозитория в указанный удалённый репозиторий
git push --mirror UrlНовогоРепозитория
Варианты
  • git push --all - отправляет только все ветки, но не теги, при этом, если в новом репозитории уже есть ветки, то они останутся
  • git push --tags - отправляет только теги
  • git push --mirror - отправляет всё (ветки, теги, рефы) и заменяет содержимое удалённого репозитория

Merge vs Rebase

Сравнение Merge и Rebase

merge rebase
История коммитов
    История коммитов нелинейная
  • Есть merge-коммиты, которые указывают сразу на 2 или больше коммитов
  • Merge не меняет историю: поверх всех коммитов, которые были в ветке, добавляется еще 1 коммит с изменениями из "develop"
    История коммитов линейная
  • Rebase меняет историю: меняются все хэши коммитов
  • Сохраняет историю коммитов
  • Не сохраняет историю коммитов
merge-коммит
  • Добавляется
  • Не добавляется
Правка конфликтов
  • Один раз
  • Для каждого коммита

Сравнение истории коммитов для Rebase и Merge

Исходные даннные: порядок коммитов
История коммитов для Rebase и Merge
  • merge (слияние) - хронологический порядок
  • rebase (перемещение) - у новых коммитов изменится хэш и они будут поверх старых
Позволяет взять все коммиты из одной ветки и применить их поверх другой
  • Выполняет работу аналогичную cherry-pick, только для цепочки коммитов, перенося ветку на новое место
  • Коммит, который не даёт изменений называется пустым. rebase автоматически пропускает этот коммит

Команды

Rebase

git rebase develop # rebase git rebase --continue # продолжить rebase (вызывается при наличии конфликтов) git rebase --skip # пропустить коммит (вызывается при наличии конфликтов) git rebase --abort # отказаться от rebase (возвращает обратно HEAD - не изменит историю коммитов) git rebase --quit # отказаться от rebase (НЕ возвращает обратно HEAD - изменит историю коммитов)
  • Флаг --continue позволяет продолжить rebase после фикса конфликта и перейти к следующему коммиту
# Rebase develop в текущую ветку git rebase develop # Если есть конфликт, то открывается редактор для фикса # После фикса добавляются изменения в staged git add . # Продолжить Rebase и перейти к следующему коммиту git rebase --continue # Откроется редактор, где можно изменить название коммита # После закрытия редактора Rebase перейдет к следующему коммиту # Если следующего коммита нет, то Rebase завершится
  • Флаг --skip позволяет пропустить коммит из текущей ветки и взять изменения из develop
# Rebase develop в текущую ветку git rebase develop # Если есть конфликт, то открывается редактор для фикса # Пропустить коммит из текущей ветки и перейти к следующему коммиту git rebase --skip
  • Флаг --abort позволяет вернусть состояние текушей ветки до rebase
  • Маркер HEAD, будет перенесен обратно на текущую, а уже добавленные коммиты повиснут в воздухе (на них не будет указывать ни один указатель) и будут вскоре удалены
# Rebase develop в текущую ветку git rebase develop # Если есть конфликт, то открывается редактор для фикса # Откатиться в состояние до rebase git rebase --abort

Интерактивный Rebase

git rebase -i develop # интерактивный rebase на другую ветку git rebase -i HEAD~3 # интерактивный rebase внутри ветки (изменить историю последних 3х коммитов)
  • При интерактивном rebase можно работать только с новыми коммитами, добавленными для текущей ветки. Пример: при git rebase -i master в ветку feature, в окне интерактивного rebase будут отражены коммиты только для feature
  • В начала rebase запустится окно редактора, где можно указать определенную команду для каждого коммита
  • При этом переписывается история коммитов - у каждого коммита будет новый хэш
› pick cb923a7 MessageCommit1 › fixup 0a101a7 MessageCommit2 › fixup 67e7534 MessageCommit3
Команды
  • pick   p - использовать коммит без изменений (по умолчанию)
  • reword r - использовать текущий коммит с возможностью изменения сообщения коммита
  • drop   d - удаление коммита (вместо этого можно удалить строку с коммитом)
  • fixup  f - будет объединен с предыдущем коммитом в списке, но без изменения сообщения коммита
  • squash s - будет объединен с предыдущем коммитом в списке
  • edit   e - отредактировать коммит (когда git дойдёт до коммита, выполнится отстановка и git предложит выполнить действия)
  • exec   x - выполнить скрипт для указанного коммита
  • break  b - остановиться здесь (продолжить rebase позже с git rebase --continue)
  • label  l - добавить метку для текущего HEAD с заданным именем
  • reset  t - сбросить HEAD до указанной метки
  • merge  m - создать merge-commit, используя message исходного merge-commit (или однострочный, если исходный merge-commit не был указан)

Алгоритм работы

Задача: загрузить изменения из master в feature

Плюсы и минусы

Примеры

  • Есть 2 ветки: develop и feature
  • Необходимо актуализаировать ветку feature: подтянуть последние изменения из develop и решить конфликты если они есть
git checkout develop # перейти в ветку develop git fetch origin # подтянуть последние изменени для develop git pull # взять последние изменени для develop git checkout feature # перейти в ветку feature
  • На выбор: делать rebase на ветку локального или удаленного репозитория
  • Если делать rebase с origin/develop, то нет необходимости делать git pull
git rebase -i develop # взять последние изменения из develop git rebase -i origin/develop # взять последние изменения из origin/develop
  • Если есть конфликты merge, откроется редактор
  • Пофиксить конфликты
git add . # добавить измененные файлы в staged git rebase --continue # если несколько коммитов, то преходим к фиксу следующего коммита
  • Если есть конфликт, то его нужно фиксить в каждом коммите
  • После фикса всех коммитов и добавления в staged (через git add), Working Tree будет чистое: не нужно ничего коммитить
git push origin HEAD # отправка изменений на удаленный репозиторий
  • Rebase изменяет историю коммитов и указатель, поэтому, если сначала сделать git push, а потом git rebase на локальном репозитории, то повторный push придется делать через push --force
  • При использовании push --force, сервер перепишет историю в ветке, которая будет повторять историю локальной ветки
git push --force origin HEAD # отправка изменений на удаленный репозиторий
Слияние веток
git merge develop # слияние "develop" в текущую ветку git merge feature develop # замёржить в "feature" из "develop" git merge --abort # отменить merge (при наличии конфликтов merge) git merge --continue # продолжить merge (при наличии конфликтов merge, после фикса) git merge --squash feature # засквошить коммиты при merge
  • При merge --squash не сработает merge --abort и merge --continue, т.к. не создается .git/MERGE_HEAD с идентификатором коммита с которым происходит слияние
  • Разрешение конфликтов разницы "develop" и "feature"
git checkout develop # перейти в develop, если в ней есть обновления git pull # обновить ветку develop git checkout feature # перейти в feature git merge develop # merge из develop в свою ветку. ветка feature станет актуальной
  • Если появятся конфликты в файлах - поправить их

Описание

  • При merge происходит создание нового merge-коммита в "feature" ветке
  • merge-коммит будет включать изменения в "develop" с момента откалывания "feature"

Плюсы и минусы

  • Merge не меняет историю: поверх всех коммитов, которые были в ветке, добавляется еще 1 коммит с изменениями из "develop". Если над "feature" веткой работают другие разработчики, то проблем не возникнет - над веткой могут работать много разработчиков
  • Сложно проследить за ветками, как они обновлялись и т.д.
  • Добавляется merge-коммит
Отмена изменений. Изменяет указатель HEAD и, опционально, состояние индекса.
  • Передвигает ветку на указанный коммит
  • Сбросить коммиты до начального в ветке
  • Убрать из индексации все файлы и директории, которые возможно уже были проиндексировалы с помощью git add
git reset HEAD~ # отменить 1 коммит git reset HEAD~1 # отменить 1 коммит git reset HEAD^ # отменить 1 коммит git reset HEAD^1 # отменить 1 коммит git reset HEAD^2 # отменить 2 коммита git reset HashCommit # откатиться к конкретному коммиту (хэш из "git log") git reset HEAD FileName # если изменения были индексированны и нужно откатиться перед самим коммитом

Виды reset

  • Каждый вид reset позволяет откатиться к указанному коммиту
git reset --soft HEAD~ # мягкий: ОСТАВИТЬ последующие коммиты git reset --hard HEAD~ # жёсткий: УДАЛИТЬ последующие коммиты git reset --mixed HEAD~ # смешанный: среднее между --soft и --hard (по умолчанию) git reset --keep HEAD~ # жёсткий с сохранением изменений git reset --merge HEAD~ # жёсткий для отмены неудачного merge
  • --soft - только перемещает ветку на указанный коммит. "working directory" и "index" без изменений
  • --mixed - перемещает ветку на указанный коммит, сбрасывает "index", но не трогакт "working directory"
  • --hard - сбрасывает "working directory" и "index" (состояние проекта переключается на указанный коммит)
Отправка изменений на удаленный git-репозиторий
git push origin HEAD # запушить текущую ветку, не вводя целиком ее название git push --force origin HEAD # запушить текущую ветку и переписать историю на сервере git push origin +feature # аналогично использованию "--force" git push origin # замерджить все ветки локального репозитория на удаленный репозиторий git push origin master # аналогично предыдущему, но делается пуш только ветки master git push --set-upstream origin master # отправка изменений на удаленный git-репозиторий git push -u origin master # указание ветки git push -f # залить на сервер измененные коммиты git push https://github.com/Account/Repo.git master # указание репозитория git push origin --delete feature # удаление ветки из удаленного репозитория

Опции

  • -f --force используется для перезаписи. позволяет обновить удаленную ссылку, которая не является предком локальной ссылки

Merge и Rebase

git pull origin master # работает по merge git pull --rebase origin master # работает по rebase
Добавление новых файлов из Working Directory в систему контроля версий (в индекс Staging Area для последующего коммита). Индексирование файлов. Сборка слепка коммита. Получить изменения из удаленного git-репозитория в локальный репозиторий
git pull origin # замерджить все ветки с удаленного репозитория git pull origin master # замержить ветку master git pull origin HEAD # замерджить текущую ветку

Варианты pull

    git pull состоит из:
  • git fetch - стягивает изменения, но не применят их
  • git merge - применяет изменения в текущую ветку. Мёржим из origin/develop в свой локальный develop

Через merge (по умолчанию)

git pull origin # или git fetch origin git merge origin/master

Через rebase

git pull --rebase origin master # или git fetch origin git rebase origin/master
Получение актуальной информации о состоянии удаленных веток (указателях) на удаланном репозитории
  • Обновление веток слежения remotes/origin
  • Стягивает изменения, но не применят их
  • Fetch скачивает все ветки с удаленного репозитория.
  • С локальными указателями веток ничего не происходит.
  • Если кто-то сделал push в ветку - в этот момент fetch скачает недостающие коммиты и передвинет ветки слежения.
git fetch --all # скачать все ветки с "origin", но не мерджить их в локальный репозиторий git fetch origin # скачать ветку "master" с "origin", но не мерджить ее в локальный репозиторий git fetch origin master

Опция --orphan

# Создание новой ветики без родительского коммита # При этом все файлы из рабочей директории будут на месте git checkout --orphan features # Создание новой пустой ветки git checkout --orphan features && git reset --hard # Создание новой ветики с определенного коммита # При этом будет создана новая ветка без родительского коммита и в индексе Git будут находится # все файлы из выбраного целевого коммита git checkout --orphan features TARGET_REFERENCE
Команда git cherry-pick используется для того чтобы взять изменения, внесённые каким-либо коммитом, и попытаться применить их заново в виде нового коммита наверху текущей ветки. Это может оказаться полезным чтобы забрать парочку коммитов из другой ветки без полного слияния с той веткой
  • Вызывается из ветки, в которую нужно добавить коммиты
git cherry-pick HashCommit1 HashCommit2

Создает метку
  • Git использует два основных типа тегов: легковесные и аннотированные
  • Легковесный тег - это что-то очень похожее на ветку, которая не изменяется - просто указатель на определённый коммит
  • Аннотированные - теги хранятся в базе данных Git как полноценные объекты. Они имеют контрольную сумму, содержат имя автора, его e-mail и дату создания, имеют комментарий и могут быть подписаны и проверены с помощью GNU Privacy Guard (GPG)
git tag v1 # создание легковесеого тега git tag -a R2021.4 -m 'Release 2021.4' # создание аннотированного тега

Сменить ветку, не теряя текущих незакоммиченных правок

Алгоритм

› git add . › git stash # сменить ветку # вернуться обратно на ветку, где был stash › git stash pop

Категории файлов для stash

По умолчанию команда git stash создает следующие отложенные изменения:

  • Изменения, добавленные в раздел проиндексированных файлов (индексированные изменения)
  • Изменения в файлах, отслеживаемых Git в настоящее время (неиндексированные изменения)

При этом следующие файлы отложены не будут:

  • Новые файлы в рабочей копии, которые еще не были проиндексированы
  • Игнорируемые файлы

Операции

Сохранение состояния рабочей директории и Индекса

  • Команда git stash сохраняет неподтвержденные изменения (индексированные и неиндексированные) в отдельном хранилище, чтобы можно вернуться к ним позже. Затем происходит откат до исходной рабочей копии
  • Отложенные изменения сохраняются в локальном репозитории Git и не передаются на сервер при выполнении команды push
git stash

Применение отложенных изменений

  • При извлечении отложенных изменений они удаляются из набора и применяются к рабочей копии
git stash pop
  • Также можно применить изменения к рабочей копии, не удаляя их из набора отложенных изменений
  • Это полезно, если нужно применить одни и те же отложенные изменения к нескольким веткам
git stash apply

Откладывание неотслеживаемых или игнорируемых файлов

"LF" и "CRLF"

При клонировании или смене ветки, Git меняет файлы с "LF" на "CRLF"

Сравнение значений

«core.autocrlf» Команды add commit База данных Git Команда checkout
«true» CRLF ➝ LF LF LF ➝ CRLF
«false» Без конвертации CRLF и/или LF Без конвертации
«input» CRLF ➝ LF LF Без конвертации

Решение

1. Проверить конфиг

git config core.autocrlf
  • Если значение отличное от false, то поменять
git config --global core.autocrlf false # глобально git config --local core.autocrlf true # для проекта

2. В проекте есть файл .gitattributes

# Auto detect text files and perform EOL normalization * text=auto
  • Файл .gitattributes имеет более высокий приоритет, чем core.autocrlf
  • http://git-scm.com/docs/gitattributes

Информация

Принцип работы

Указатели

  • HEAD - ссылка на последний коммит и на текущую ветку ветку
  • Branch - ветка-указатель на коммит. На один коммит может быть несколько веток-указателей (н-р: merge-commit)
  • remotes/origin/master - указатель на каком коммите находится указатель ветки в удаленном репозитории. Ветки слежения удаленного репозитория
  • Локально origin нет

Статусы

  1. Untracked - после добавления нового файла
  2. Staged - после git add
  3. Modified - после изменения существующего файл
  4. Commited - после git commit

Правила

  • Каждый коммит ссылается на предыдущий. В качестве ссылки выступает хэш-сумма коммита

Local Operation

  • I. Working Directory - рабочий каталог
    • 1. Внесение изменений в файлы рабочего каталога
    • 2. Подготовка файлов, добавляя из слепки в область Staging Area
  • II. Staging Area - область подготовленных файлов
    • 3 commit. Берет подготовленные файлы из индекса Staging Area и помещает их в Git Directory на постоянное хранение
  • III. Git Directory (Repository) - каталог git

File Status Lifecycle

  • Файл в Working Directory может находиться под версионным контролем (unmodified, modified, staged) и неотслеживаемые файлы (untracked)

Gitflow

  • Gitflow - модуль управления ветками. Есть две основные ветки: master и develop
  • gitflow-avh - инструмент для работы с Gitflow

Ветки

  • master - содержится тот же код, что и в рабочей (продакт) версии проекта
  • release - подготовка к новому релизу проекта
  • beta - beta
  • develop - происходит вся работа
  • feature - создаются в процессе работы над develop, их может быть неограниченное количество
  • improvement - улучшение
  • bug - исправление багов
  • bugfix - исправление багов
  • hotfix - срочное исправление багов, найденных, например, на продакте
  • fix - fix

Рабочий процесс в Gitflow

  1. Создается репозиторий
  2. Репозиторий инициализируется
  3. Начинается работа на ветке develop
  4. Возникает необходимость опробовать новую штуку – создается feature-ветка и делаются коммиты
  5. Закончив работу на feature-ветке, вы сливаете ее с develop
  6. Если вы довольны текущей версией, но хотите продолжить работу, создается ветка release, куда перемещается текущая версия. Правка багов будет происходить на этой же ветке
  7. Когда с веткой release покончено, время слить ее в master и продолжить работу с develop
  8. Кроме того, этот момент можно отметить на master-ветке

Codereview GitHub

  1. Создать репозиторий
  2. Основная ветка разработки "dev"
  3. Под каждую задачу создается ветка от "dev"
  4. Создание PR (Pool Request) для слития ветки задачи с веткой "dev"
  5. Ссылка на PR руководителю
  6. Если всё ок, слитие веток, если нет, то исправление замечания
  7. Если несколько задач и несколько веток, то вливать нужно не ветки полностью а только отдельные коммиты

Системы управления версиями

Типы

  • Централизованная система управления версиями (Version Control System, VCS или Revision Control System) - позволяет хранить несколько версий одного и того же документа, при необходимости возвращаться к более ранним версиям, определять, кто и когда сделал то или иное изменение, и многое другое
  • Такие системы наиболее широко используются при разработке программного обеспечения для хранения исходных кодов разрабатываемой программы
  • Традиционные системы управления версиями используют централизованную модель, когда имеется единое хранилище документов, управляемое специальным сервером, который и выполняет бо́льшую часть функций по управлению версиями
  • Контроль версий означает что вы храните все версии редактируемых документов и можете вернуться к любой сохраненной версии в любой момент времени
  • Примеры: SVN, Team Foundation Server
  • Распределённые системы управления версиями (Distributed Version Control System, DVCS) - используют распределённую модель вместо традиционной клиент-серверной. Они, в общем случае, не нуждаются в централизованном хранилище: вся история изменения документов хранится на каждом компьютере, в локальном хранилище, и при необходимости отдельные фрагменты истории локального хранилища синхронизируются с аналогичным хранилищем на другом компьютере. В некоторых таких системах локальное хранилище располагается непосредственно в каталогах рабочей копии
  • Когда пользователь такой системы выполняет обычные действия, такие как извлечение определённой версии документа, создание новой версии и тому подобное, он работает со своей локальной копией хранилища. По мере внесения изменений, хранилища, принадлежащие разным разработчикам, начинают различаться, и возникает необходимость в их синхронизации. Такая синхронизация может осуществляться с помощью обмена патчами или так называемыми наборами изменений (англ. change sets) между пользователями.
  • С точки зрения пользователя распределённая система отличается необходимостью создавать локальный репозиторий и наличием в командном языке двух дополнительных команд: команды получения репозитория от удалённого компьютера (pull) и передачи своего репозитория на удалённый компьютер (push). Первая команда выполняет слияние изменений удалённого и локального репозиториев с помещением результата в локальный репозиторий; вторая — наоборот, выполняет слияние изменений двух репозиториев с помещением результата в удалённый репозиторий
  • Примеры: Git, Mercurial

Общее

  • Git - распределенная система контроля версий (version control system - VCS)

Git Submodules

  • Запись, которая указывает на commit в другом внешнем репозитории
  • Подмодули очень статичны и отслеживают только определенные коммиты
  • Подмодули не обновляются автоматически при обновлении основного репозитория
