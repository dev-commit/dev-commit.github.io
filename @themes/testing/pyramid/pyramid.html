Пирамида тестирования

  • Нижняя часть - это самые быстрые, простые и самые изолированные тесты
  • Верхняя часть - самые ресурсоемкие, самые медленные и охватывают всё приложение в целом

Виды тестов

  • 1 Unit Testing (Блочное)
  • Тестирование одного модуля в изоляции
  • 2 Integration Testing (Интеграционное)
  • Тестирование группы взаимодействующих модулей
  • 3 System Testing (Системное)
  • Тестирование системы в целом
  • Для небольших проектов это, как правило, ручное тестирование - запустил, пощелкал, убедился, что (не) работает. Можно автоматизировать
  • 4 E2E Testing
  • Тестирование UI

System Testing (2 подхода к автоматизации)

  1. Использовать вариацию MVC паттерна - Passive View и формализовать взаимодействие пользователя с GUI в коде. Тогда системное тестирование сводится к тестированию Presenter классов, а также логики переходов между View
  2. Использовать специальные инструменты для записи действий пользователя. То есть в итоге запускается сама программа, но щелканье по кнопкам осуществляется автоматически

Информация

Google часто предлагает разделение 70/20/10

  • 70% unit тестов
  • 20% интеграционных тестов
  • 10% E2E тестов

Информация

  1. Модульные тесты реализовывать быстрее, чем GUI тесты (время на разработку меньше). Модульные тесты отрабатывают быстрее, чем GUI
  2. Модульные тесты дешевле, чем GUI тесты (однако, стремясь к вершине пирамиды, мы получаем большую уверенность в том, что все работает как ожидалось)
  3. Тестирование должно быть разносторонним, но соотношение тестов должно быть таково, что модульных количественно больше, чем GUI

Принципы

  1. Писать тесты разной детализации
  2. Чем выше уровень, тем меньше тестов
